# Prompt 增强原则与技巧

## 核心理念

好的 prompt 应该具备 **SMART** 特性：
- **S**pecific（具体）：明确说明要做什么
- **M**easurable（可衡量）：有清晰的完成标准
- **A**chievable（可实现）：在当前上下文中可执行
- **R**elevant（相关）：与项目/需求直接相关
- **T**estable（可测试）：结果可验证

## 增强维度

### 1. 具体化模糊词汇

| 模糊表述 | 增强方向 |
|---------|---------|
| "make it good" | 定义"好"的标准：性能指标、代码质量、用户体验 |
| "fix the bug" | 描述 bug 现象、复现步骤、预期行为 |
| "add a feature" | 详细功能点、交互流程、边界情况 |
| "improve performance" | 具体指标：加载时间、响应速度、内存占用 |
| "clean up code" | 重构目标：减少重复、提取函数、改善命名 |

### 2. 补充上下文

必要的上下文信息：
- **技术栈**：使用什么语言/框架
- **约束条件**：兼容性要求、性能预算
- **现有代码**：相关文件位置、依赖关系
- **用户场景**：谁会用、怎么用

### 3. 定义验收标准

好的验收标准：
- ✅ "用户可以通过点击按钮切换主题"
- ✅ "API 响应时间 < 200ms"
- ✅ "测试覆盖率 > 80%"

差的验收标准：
- ❌ "代码质量好"
- ❌ "用户体验流畅"
- ❌ "性能优秀"

### 4. 识别隐含需求

常见的隐含需求：
- 错误处理和边界情况
- 安全性考虑（输入验证、权限检查）
- 可访问性（a11y）
- 国际化（i18n）
- 响应式设计
- 日志记录

## 分类处理策略

### 新功能开发
重点增强：
- 功能边界和范围
- 数据模型设计
- API 接口设计
- UI/UX 规范
- 测试策略

### Bug 修复
重点增强：
- 问题复现步骤
- 预期 vs 实际行为
- 影响范围
- 修复验证方法

### 性能优化
重点增强：
- 当前性能基线
- 目标性能指标
- 可接受的权衡
- 度量方法

### 代码重构
重点增强：
- 重构范围限定
- 保持行为不变的约束
- 质量指标改进
- 测试保障

## 常见问题处理

### 需求过于宽泛
策略：分解为多个独立任务

```
原始：Build an e-commerce website
增强：分解为
1. 用户认证模块
2. 商品目录页面
3. 购物车功能
4. 结账流程
5. 订单管理
```

### 需求信息不足
策略：列出需要澄清的问题

```
原始：Add a form
增强：需要澄清
- 表单用途（注册/联系/调查）？
- 需要哪些字段？
- 提交后的处理逻辑？
- 是否需要验证？
```

### 需求存在歧义
策略：列出可能的解读，让用户选择

```
原始：Make the button bigger
增强：可能的解读
A) 增大按钮尺寸（padding/font-size）
B) 增加按钮的视觉权重（颜色/阴影）
C) 让按钮更容易被注意到（位置/动画）
请确认你的意图。
```

## 质量检查清单

增强后的 prompt 应该能回答：
- [ ] 要做什么？（任务描述）
- [ ] 用什么做？（技术要求）
- [ ] 怎么做？（实现细节）
- [ ] 做成什么样？（完成标准）
- [ ] 需要注意什么？（约束和注意事项）
